## 1. Les quatre domaines sont devenus quatre schémas — plus un cinquième

`db/00_roles_schemas.sql` l. 37-41 :

```sql
CREATE SCHEMA identite AUTHORIZATION continuum_migration;   -- domaine 0
CREATE SCHEMA recolte  AUTHORIZATION continuum_migration;   -- domaine 1
CREATE SCHEMA depot    AUTHORIZATION continuum_migration;   -- domaine 2
CREATE SCHEMA machine  AUTHORIZATION continuum_migration;   -- domaine 3
CREATE SCHEMA lecture  AUTHORIZATION continuum_migration;   -- la fenêtre 0-1-2 → machine
```

Le domaine n'est plus une idée : c'est **la frontière de nommage**. Et un cinquième schéma,
`lecture`, où vivent les **vues** par lesquelles la machine lit 0-1-2 — *« la fenêtre, pas la
porte »* (`db/50` l. 1). Ces vues appartiennent à la migration, ne sont pas
`security_invoker` : elles s'exécutent avec les droits du propriétaire, donc *« lire à
travers la fenêtre n'exige aucun droit sur la table de base — et n'en donne aucun »*
(`db/50` l. 14-16). Le chemin de lecture existe **sans** que le chemin d'écriture existe.
C'est la Vigilante rendue en SQL : on lit, on ne dépose pas, faute d'opération pour le faire.

## 2. L'invariant 2 passe de « structure » à **absence**

`schema.md` l. 533, réécrit :

> | 2 | La machine n'écrit jamais un dépôt | **absence** — rôle machine sans `USAGE` sur le
> schéma `depot` : le nom ne se résout pas. *Un `REVOKE INSERT` laisse la table nommable,
> donc un `GRANT` la rouvre : ce serait une permission, non une absence.*

Tenu dans `db/60_privileges.sql` l. 36 — la **totalité** de ce que la machine peut nommer :

```sql
GRANT USAGE ON SCHEMA lecture, machine TO continuum_machine;   -- ni identite, ni recolte, ni depot
```

**La preuve est dans deux messages d'erreur distincts**, tous deux `42501`, mais pas sur le
même objet :

| test | geste | détail imprimé |
|---|---|---|
| 1 | la machine insère un dépôt | `42501 — permission denied for **schema** depot` |
| 3 | un `UPDATE` sur un dépôt (par le soignant, qui, lui, y écrit) | `42501 — permission denied for **table** depots` |

Au test 1, la machine ne se voit pas *refuser l'écriture* : elle ne trouve pas le guichet, le
nom `depot.depots` ne se résout pas (`db/tests` l. 124-126). Au test 3, la table existe et se
nomme — c'est le droit d'`UPDATE` qui manque (invariant 1). *Refusé parce que le nom n'existe
pas* et *refusé parce que le CHECK/le droit a mordu* sont deux mondes (`db/tests` l. 42-44).

Et pas davantage par la fenêtre : test 1 bis, `55000 — cannot insert into view "depots"` — la
vue porte un `WITH` de tête, non auto-inscriptible, *« aucun GRANT ne la rouvre »* (`db/50`
l. 5-7).

## 3. Loi §0 n°4, neuve : **la clé se pose, l'attribut se corrige**

`schema.md` §0 l. 38 :

> **La clé se pose ; l'attribut se corrige.** Corriger, c'est insérer une ligne qui référence
> la précédente : un attribut corrigible ne peut donc **pas** être une clé primaire (deux
> lignes, un IPP). […] Ce n'est pas une commodité d'implémentation : c'est la loi §0 qui
> l'exige.

Le schéma d'avant écrivait `patients_identite (ipp text PRIMARY KEY, …, corrige_id uuid)` :
**impossible**. Corriger insère une ligne portant `corrige_id` — donc deux lignes pour un
IPP, ce que `ipp PRIMARY KEY` interdit (`db/10` l. 3-13). L'append-only **force** la coupe :

- `identite.patients` — la clé nue (`ipp text PRIMARY KEY`, `db/10` l. 18-21) ;
- `identite.patients_identite` — la fiche chaînée, `corrige_id` → la ligne corrigée (l. 25-41).
- Même coupe sur l'agent (`db/20` l. 14-33) : `recolte.agents` (clé nue) / `recolte.agents_fiche`
  (chaînée). La raison est nommée l. 10-12 : *« `auteur_id` d'un dépôt doit pointer vers un
  identifiant stable pour toujours ; un dépôt est immuable, et l'agent change de fonction. »*

La chaîne est tenue par la **structure**, pas par une consigne : FK composite
`(corrige_id, ipp) → (id, ipp)` (une correction ne peut pas faire migrer une identité d'un IPP
à l'autre, l. 39-40), plus deux index partiels uniques — **une seule tête par IPP** (l. 44-45)
et **une seule correction par ligne corrigée** : *« une chaîne, jamais un arbre »* (l. 48-49).

## 4. Le troisième cadre — `responsabilite_medicale` — et la coupe **signé / formulé**

`db/30_domaine_2_depots.sql`. Trois cadres, jamais trois grades (l. 47-55) : le cadre nomme
**le moment**.

```sql
CONSTRAINT depots_cadre_connu
  CHECK (cadre IN ('seul','synthese_collective','responsabilite_medicale')),
```

La coupe n'est pas entre le médecin et les autres, mais entre ce qui se **signe** et ce qui
se **formule** (l. 67-83) :

- **Ce qui se formule — la grille** (`situation`, `ressenti`, `demande`, `diffraction`,
  `equilibre`) — est **fermé à tout le monde, le coordinateur compris** (`db/30` l. 88-90) :
  ```sql
  CONSTRAINT depots_grille_formulee_en_college
    CHECK (nature NOT IN ('situation','ressenti','demande','diffraction','equilibre')
           OR cadre = 'synthese_collective'),
  ```
  *« `diffraction` déposée seul est un homme qui se relit ; `equilibre` déposé seul est le
  nouage rendu à un regard unique. Si l'on peut formuler seul, le trilobe cesse d'être un
  collège : c'est un bureau »* (l. 74-78). Prouvé deux fois : test 4 (une IDE), test 5 (**le
  coordinateur lui-même**), tous deux `23514 — check_violation`.
- **Ce qui se signe** (`lecture_clinique`, `compte_rendu`, `validation_typage`) engage une
  responsabilité, donc une personne — ouvert au coordinateur seul, sous `responsabilite_medicale`
  (l. 92-100) : `depots_signe_jamais_seul` + `depots_responsabilite_medicale_reservee`
  (*« ce cadre n'est pas une porte de service »*). Prouvé : tests 11-12 (succès), et le cadre
  **s'inscrit dans le record**, daté et visible.

La **permission applicative** de l'ancien invariant 13 était inimplémentable — dit en clair
dans `schema.md` l. 544 : *« une permission applicative ne peut que restreindre : elle
n'accorde jamais au-delà d'un `CHECK`. L'exception devait donc s'écrire, pas se contourner. »*

## 5. L'attribution tenue à la racine — `auteur_id = agent_connecte()` par RLS

Un `CHECK` dit *ce qui* passe ; il ne peut pas dire *qui* (il ne regarde qu'une ligne). Le
droit de fonction est donc une **politique de ligne** (`db/30` l. 183-223) :

```sql
CREATE POLICY depots_depot_attribue ON depot.depots
  FOR INSERT TO continuum_soignant
  WITH CHECK (
    auteur_id = depot.agent_connecte()
    AND (cadre <> 'responsabilite_medicale' OR depot.est_coordinateur(auteur_id))
  );
```

La première condition n'est pas un ornement : *« sans elle, écrire l'uuid du coordinateur
dans `auteur_id` suffirait à emprunter son droit — la garde serait décorative »* (l. 189-191).
C'est **l'invariant 18 tenu à la racine, non à l'écran** : nul ne dépose sous le nom d'un
autre. Prouvé au test 7 (`42501` — Karima dépose sous l'uuid du médecin). `est_coordinateur()`
lit la **queue** de la chaîne de fiches (l. 197-203) : le droit suit la fonction, qui change,
jamais la personne.

**Conséquence dure pour l'app** : `SET LOCAL continuum.agent_id = '…'` à l'ouverture de chaque
transaction (`agent_connecte()` lit ce réglage, l. 205-208). Sans lui, `auteur_id` ne peut
égaler personne : **la base refuse un écrit anonyme.**

## 6. `ref_id` scindé ; `type_valide` sorti de la prose

`ref_id` était **polymorphe** dans le schéma — *« dépôt révisé / levé, **ou** proposition
validée »* : deux tables, deux domaines, une colonne, donc aucune FK possible, donc un uuid
nu, **falsifiable** — la chose même que la loi 3 bannit (`db/30` l. 8-13). Scindé :

- `ref_depot_id` (domaine 2) + `ref_nature`, sous **FK composite non falsifiable**
  `(ref_depot_id, ipp, ref_nature) → (id, ipp, nature)` (`db/30` l. 45, 151-153). Elle achète
  en structure deux gardes qui seraient sinon du code : *une levée ne lève qu'une `hypothese`,
  une `inquietude` ou une `gestation`* (`depots_levee_reference`, l. 140-143) ; *une révision
  révise le même registre* (`depots_revision_homogene`, l. 145-148).
- `ref_proposition_id` (domaine 3), FK vers `machine.propositions_retypage` posée en `db/40`
  l. 27-29 : elle n'y lit **aucune vérité** (une proposition reste une proposition) — elle
  atteste *qu'un acte machine daté a eu lieu, et lequel*.

`type_valide` **sort du `contenu`** dans sa propre colonne (l. 41, 155-159). Rangé dans la
prose, la projection `type_courant` obligerait la machine à **lire la phrase** — la
comparaison sémantique sur la surface vive que §4 (d) rejette (l. 19-22). *« Ce n'est pas un
nombre. »* Et l'exception *« sauf premier typage »* devient **structurelle** : un index unique
partiel, *« au plus un typage sans proposition par patient »* (l. 167-169) — au lieu d'un
« premier » que nul ne sait définir en SQL.

## 7. La seule garde qui reste du code — à nommer pour qu'elle ne se dissolve pas

`db/60_privileges.sql` l. 9-17, et `schema.md` §11 l. 4 (*« tranché en Phase 0 »*) :

> Le propriétaire d'une table conserve tous les droits sur elle, et un superutilisateur passe
> au travers de tout. Les deux invariants \[1 et 2] valent donc **exactement aussi longtemps
> que la discipline de connexion** : l'application se connecte comme `continuum_soignant` ou
> `continuum_machine`, jamais comme `continuum_migration`, jamais comme superutilisateur.

C'est la dernière garde **de code, non de structure**. Aucune DDL ne peut la tenir — une DDL
qui « tient » est justement le confort où ce point se dissoudrait. Nommée ici pour qu'on ne
l'oublie pas. (Corollaire vérifié au montage : le rôle superutilisateur `postgres` que j'avais
créé pour lancer les tests a été **supprimé** ; le cluster n'a plus que le compte de la
machine, et `run_db_test.sh` tourne sur ses défauts — `PGPORT=5432`, `PGUSER=$(whoami)`.)

## Ce qui reste faux — et qu'on ne corrige PAS ici

`CONTINUUM.md` §10 dit **encore** que l'exception du coordinateur est « à régler dans le
système de permissions ». C'est faux depuis ce matin : elle ne se règle pas dans une
permission (qui ne peut que restreindre), elle se pose dans un **cadre** — `responsabilite_medicale`,
inscrit au record (§4 ci-dessus). Mais **la doctrine grave après que le schéma est tranché**,
jamais l'inverse : cette rectification vit dans un autre fil, pas dans Phase 0. Signalée, non
corrigée.

*(Deux « écarts assumés avec `schema.md` » que les DDL portaient en tête — `db/10` l. 3-14,
`db/30` l. 8-23 — sont désormais **résolus** : `schema.md` a été aligné en `a6779c8`. Ils ne
sont plus des écarts ; ils sont l'histoire de la coupe.)*

## Prochain geste — l'UI

Une **zone de saisie**, un **bouton**, et **aucune validation de rangement**. Il n'y a plus de
colonne `axes` sur un dépôt (`db/40` l. 82 : *« pas de table `fragments`, pas de colonne
`axes` »* ; invariant 20), donc plus d'écran de tri à valider : les fragments se **recalculent**
à côté du texte, ils ne se gravent pas. Le soignant écrit, ça s'écrit — `SET LOCAL
continuum.agent_id`, un `INSERT` dans `depot.depots`, rien de plus.

Le premier écran qui **refuse une donnée** tranchera les points encore ouverts mieux que la
prose : *si un champ réclame une glose, c'est le champ qui est mal fait* (§8). La base, elle,
refuse déjà — anonyme, seul-quand-il-faut-être-plusieurs, machine devant le sanctuaire. L'UI
n'a qu'à ne pas mentir sur ce que la base tient.
